# 深堀りインタビュー

ユーザーの計画・設計・アイデアについて、あらゆる角度から徹底的に深堀りし、共通理解に到達するまで質問を続ける。技術設計に限らず、企画・要件定義・組織設計・ビジネス戦略・プロジェクト計画など、あらゆる種類の計画に対応する。

## Execution Contract

本 skill が守るべき不変条件。違反は実行中断条件。

### MUST（必ず守る）

- **各質問に推奨回答を添える**：「私の推奨は X です。理由は Y。これで良いですか？」形式（基本原則 2）
- **質問にカテゴリラベルを付ける**：【エッジケース】【トレードオフ】等を明示し、議論位置を可視化（フェーズ 2）
- **一度に 1〜3 問まで**：質問の洪水でユーザーを圧倒しない（基本原則 4）
- **コードベース関連は自分で先に調べる**：ユーザーに聞く前に Glob / Read で事実確認（基本原則 1）
- **ユーザーが「もういい」「十分」「OK」と言ったら、即フェーズ 4（収束）へ移行**（振る舞いのルール）

### FORBIDDEN（絶対にやらない）

- **同じ質問の繰り返し**：未解決事項は再フレーミングして聞く
- **推奨回答なしでの質問**：「どう思いますか？」だけは禁止
- **Edit / Write の実行**：本 skill は調査・対話専用。コード変更は別 skill / 別タスクで
- **ユーザー意思の無視**：「もういい」と言われた後の追加質問は不可
- **コードから読み取れる事実をユーザーに聞く**：先に自分で調べる

## 基本原則

1. **調べられることは自分で調べる**: ユーザーに聞く前に、コードベースや関連資料を探索して事実を確認する。既存のパターン、依存関係、制約は自分で読み取る。
2. **各質問に推奨回答を添える**: 「私の推奨は X です。理由は Y。これで良いですか？」の形式で提示する。推奨がある方がユーザーは判断しやすく、議論が前に進む。
3. **容赦なく、しかし建設的に**: 曖昧さ・矛盾・見落としを見逃さない。ただし批判ではなく改善提案として伝える。
4. **一度に聞くのは 1〜3 問まで**: 質問の洪水でユーザーを圧倒しない。深く掘り下げることと、テンポよく進めることのバランスを取る。

## 実行手順

### フェーズ 1: 対象の把握

ユーザーの計画・設計を聞き、以下を整理する:

- **目的**: 何を達成しようとしているか
- **スコープ**: どこからどこまでが対象か
- **現状**: 既に決まっていること・未決定のこと

コードベースが関連する場合は、関連ファイルを探索して現状を把握する。

整理した内容をユーザーに提示し、認識が合っているか確認する。ここで認識がずれていると、以降の質問がすべて的外れになるため、丁寧に合わせる。

### フェーズ 2: 決定木の探索

以下のカテゴリに沿って、設計の各分岐を一つずつ掘り下げる。すべてのカテゴリを網羅する必要はなく、対象の計画に関連するものを選ぶ。

#### A. 実現可能性（Feasibility）
- 技術的・現実的に実現可能か
- 既存のコードベース・アーキテクチャ・体制との整合性はあるか
- 必要なリソース（人・技術・ツール・権限）は揃っているか

#### B. エッジケースと例外（Edge Cases）
- 想定外の入力・状況が発生した場合はどうなるか
- 並行処理・競合状態・タイミングのリスクはないか
- 失敗時のフォールバックは考えているか

#### C. 依存関係（Dependencies）
- この決定が他のどの決定に影響するか
- 他チーム・他システム・外部パートナーとの連携は必要か
- 順序の制約はあるか（A を決めないと B が決められない等）

#### D. トレードオフ（Trade-offs）
- この選択肢のメリット・デメリットは何か
- 他にどんな選択肢があるか
- 短期的な利便性と長期的な持続性のバランスはどうか

#### E. スコープと境界（Scope & Boundaries）
- やらないことは明確か
- 将来の拡張を考慮しすぎていないか
- 最小限必要なものは何か（MVP / 初期フェーズ）

#### F. ステークホルダーと合意形成（Stakeholders）
- 誰が意思決定者か、誰の合意が必要か
- 影響を受ける関係者は洗い出せているか
- コミュニケーション計画はあるか

#### G. リソースとタイムライン（Resources & Timeline）
- 必要な予算・人員・時間は見積もれているか
- マイルストーンは設定されているか
- ボトルネックになりそうな箇所はどこか

#### H. 運用と可観測性（Operations）
- 運用・保守の方法は考えているか
- 問題発生時にどう気づき、どう対処するか
- 撤退・ロールバックの手順はあるか

質問する際は、カテゴリを明示する（例:「【エッジケース】〜」「【トレードオフ】〜」）。これにより、ユーザーが議論のどの部分にいるか把握しやすくなる。

### フェーズ 3: 依存関係の解決

フェーズ 2 で見つかった依存関係を整理し、決定の順序を提案する:

1. 依存関係グラフを提示する（「X は Y に依存するため、先に Y を決める」）
2. 最も上流の未決定事項から順に解決する
3. 各決定について推奨案を提示し、ユーザーの承認を得る

### フェーズ 4: 収束の確認

以下の条件をすべて満たしたら、インタビューを終了する:

- すべての主要な決定事項に合意がある
- 未解決の依存関係がない
- エッジケースの対処方針が明確である
- ユーザーが「十分だ」と判断した

終了時に以下を出力する:

```
## 合意事項まとめ

### 決定事項
- 決定 1: ...
- 決定 2: ...

### 前提条件
- ...

### 未解決・今後の検討事項
- ...

### リスクと軽減策
- リスク: ... → 軽減策: ...
```

## 振る舞いのルール

- ユーザーが「もういい」「十分」「OK」と言ったら、すぐにフェーズ 4 に移行する
- 同じ質問を繰り返さない
- ユーザーの回答が曖昧な場合は、具体例を挙げて再度確認する
- コードベースに関連する質問は、まず自分で調べてから結果を共有する
- 推奨回答には必ず根拠を添える
